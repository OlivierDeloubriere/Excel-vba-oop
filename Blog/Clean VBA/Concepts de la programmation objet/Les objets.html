Les objets

Une classe définit un objet

Une classe permet de définir ce que doit être un objet : une maison, une voiture, une interface utilisateur, une facture, une base de données ou tout autre concept qui apparaît utile dans la conception de notre logiciel.

Par exemple, grâce à la définition de la classe Facture, nous pouvons créer dans l'exécution de notre code autant de factures que nous voulons, toutes construites selon le même modèle de classe. C'est pratique. La création d'un objet selon le modèle d'une classe s'appelle une instanciation (ou une instance) de la classe.

En VBA, on définit une classe d'objet dans un module de classe.

Un objet possède des champs, des propriétés et des méthodes

Les objets peuvent avoir des variables spécifiques

Un champ, Field , est une variable qui appartient à l'objet. Un champ peut avoir n'importe quel type : entier, réel, booléen ou tout autre objet...

En VBA un champ est simplement une variable définie au niveau du module de classe.

Les objets peuvent avoir des fonctions spécifiques

Une méthode est une fonction pouvant être appelée au niveau de l'objet. Par exemple démarrer la voiture est une méthode de l'objet voiture.

En VBA, comme dans le code des modules standards, les méthodes définies dans les classes sont soit des fonctions Function (qui retournent une valeur), soit des procédures Sub (fonctions qui ne retournent pas de valeur).

Une classe expose ses champs et ses méthodes Publiques

L'ajout du mot clé Public devant la définition d'un champ, d'une fonction ou d'une sub permet d'accéder à ces derniers depuis l'extérieur de la classe. Si l'on considère que certains champs ou méthodes ne doivent pas être accessibles à l'extérieur on utilise le mot clé Private.

La bonne pratique : tous les champs doivent être privés

Pour accéder à un champ privé de la classe depuis l'extérieur, nous définissons des Public Property. Les propriétés publiques de la classe exposent les champs privés de la classe à l'extérieur.

Deux type de propriétés existent :

  • Property Get qui permet d'extraire la valeur du champ privé
  • Property Let (ou Set s'il s'agit d'un objet) qui permet d'attribuer une valeur au champ privé

En VBA, la syntaxe de définition d'un champ et de ses propriétés Get et Let se fait comme suit :

Private _maVariableEntiere As Integer
Private _maFeuille As Worksheet

Public Property Get maVariableEntiere() As Integer
    maVariableEntiere = _maVariableEntiere
End Property

Public Property Let maVariableEntiere(byVal value as Integer)
    _maVariableEntiere = value
End Property
    
Public Property Get maFeuille() As Worksheet
    Set maFeuille = _maFeuille
End Property

Public Property Set maFeuille(byVal value as Worksheet)
    Set _maVariableEntiere = value
End Property
Private _maVariableEntiere As Integer
Private _maFeuille As Worksheet

Public Property Get maVariableEntiere() As Integer
    maVariableEntiere = _maVariableEntiere
End Property

Public Property Let maVariableEntiere(byVal value as Integer)
    _maVariableEntiere = value
End Property
    
Public Property Get maFeuille() As Worksheet
    Set maFeuille = _maFeuille
End Property

Public Property Set maFeuille(byVal value as Worksheet)
    Set _maVariableEntiere = value
End Property

Notons que nous faisons précéder par un underscore le nom du champ privé, afin de le différencier de la propriété publique. D'autres conventions existent comme faire précéder les noms des champs privés par la lettre p. Nous verrons au chapitre suivant comment faire pour utiliser le même nom de variables à la fois pour les champs privés et les propriétés qui les exposent.

Le reste du code fait appel aux classes

En VBA on peut écrire du code dans des modules standards et dans des modules de classe. On peut tout à fait concevoir des outils en VBA sans jamais utiliser de classes. Enfin... On est souvent amener à écrire des choses comme

Worksheet("Ma Feuille").Range("DébutListe").Value
Worksheet("Ma Feuille").Range("DébutListe").Value

En réalité on utilise ici la classe Worksheet instanciée en "Ma Feuille" qui possède une propriété Range qui renvoie un objet plage de cellules ayant lui-même une propriété Value qui renvoie la valeur de la cellule.

L'écriture dans les modules standards est spécifique au VBA et nous sort du paradygme de la programmation orienté objet (pour revenir à au paradygme de la programmation procédurale). On peut cependant tenter de limiter au maximum l'écriture de code dans les modules standards et de spécifier le plus de classes possibles. Dans ce cadre nous pouvons mettre à profit les excellentes pratiques de conception et d'architecture logicielles de la programmation orientée objet. Oui mais attention : il existe peu de bonnes pratiques et hélas, beaucoup de mauvaises. Concevoir de bonnes architectures logicielles orientées objet est une pratique qui s'apprend.

Il est à noter que la notion de module standard et de module de classe est une spécificité du VBA qui vient encore une fois de son IDE. Mais les langages comme C# ou Python, pourtant très fortement orientés objet, permettent de définir également des fonctions et des procédures sans avoir à les encapsuler dans une classe. C'est ce qu'on appelle des fonction de haut niveau (high level functions). En revanche c'est interdit en Java, tout code doit être écrit dans une classe.

Deux concepts fondamentaux : l'héritage et l'implémentation

Héritage de classe

Dans la plupart des langages orientés objet, une classe peut hériter d'une autre classe. Cela permet par exemple de définir une classe Animal générique puis de définir des classes Chat et Canard qui héritent de la classe Animal et possèdent donc ses champs, propriétés et méthodes (par exemple respire). Les deux classes Chat et Canard peuvent ensuite contenir des propriétés et méthodes spécifiques à elles-mêmes (par exemple vomitSesCroquettes pour la classe Chat et voleEnFormation pour la classe Canard).

C'est très pratique MAIS un des défauts principaux de l'héritage est la création d'une arborescence de plus en plus grande et profonde de classes héritant les unes des autres. C'est un piège dans lequel il vaut mieux ne pas tomber car une telle arborescence est difficilement maintenable et évolutive.

Implémentation d'interface

Une interface est une classe dont la finalité n'est pas d'être instanciée mais de servir de patron de conception pour d'autres classes. Dans une interface, on définit les propriétés et méthodes communes, sans écrire le code spécifique qui irait à l'intérieur. Toute classe implémentant cette interface doit avoir les propriétés et méthodes définies dans l'interface mais le code qu'on y écrit peut être spécifique à notre classe. C'est très puissant : définissons par exemple une interface DataTable avec la méthode publique GetData :

Public Function GetData() As Variant
    'Ici on n''écrit rien !
End Function
Public Function GetData() As Variant
    'Ici on n''écrit rien !
End Function

Tout code client faisant appel à des objets de type DataTable peut invoquer la méthode publique GetData qui renvoie un Variant. Maintenant définissons deux classes : WorksheetDataTable et MySQLDataTable, chacune implémentant cette interface. En VBA il suffit d'écrire dans le module de classe : Implements DataTable. Le code ne compilera pas tant que ces deux classes n'ont pas de propriété Data. En VBA la syntaxe est imposée pour implémenter les propriétés et méthodes de notre interface parente :

Private Function DataTable_GetData() As Variant
    'Ici on écrit le code qui retourne le variant avec les données souhaitées'
End Function
Private Function DataTable_GetData() As Variant
    'Ici on écrit le code qui retourne le variant avec les données souhaitées'
End Function

Notez qu'ici la fonction est privée et que le nom de l'interface doit être inclus avec un underscore.

Rien n'empêche à présent d'écrire du code spécifique à chacune de nos classes pour renvoyer les données issues d'une table :
Dans WorksheetDataTable.cls :

Private Function DataTable_GetData() As Variant
    DataTable_GetData = GetDataFromWorksheet(worksheetName, rangeName)
End Function
Private Function DataTable_GetData() As Variant
    DataTable_GetData = GetDataFromWorksheet(worksheetName, rangeName)
End Function

et dans MySQLDataTable.cls :

Private Function DataTable_GetData() As Variant
    DataTable_GetData = GetDataFromMySQLDataBase(connexionString, tableName)
End Function
Private Function DataTable_GetData() As Variant
    DataTable_GetData = GetDataFromMySQLDataBase(connexionString, tableName)
End Function

Le code client qui a besoin d'extraire les données soit d'une feuille soit d'une base de données peut travailler avec un objet de type DataTable (notre interface) et faire appel à sa méthode GetData. L'instanciation réelle de l'objet de type DataTable peut être défini comme une instanciation de la classe WorksheetDataTable ou de la classe MySQLDataTable, peu importe, le code sera le même.

En pratique voici un exemple de ce que le code client pourrait implémenter :

Public Sub ReadAndWriteData(dataSource as DataTable)
    'Ici la variable dataSource est du type de notre interface
    Dim data as Variant
    data = dataSource.GetData
    WriteData(data)
End Sub
Public Sub ReadAndWriteData(dataSource as DataTable)
    'Ici la variable dataSource est du type de notre interface
    Dim data as Variant
    data = dataSource.GetData
    WriteData(data)
End Sub

Ce code s'en fiche de savoir si dataSource est un objet WorksheetDataTable ou un objet MySQLDataTable.

Bien entendu dans la partie du code qui s'occupe de récupérer les données à partir d'une feuille, on instanciera dataSource en tant qu'objet WorksheetDataTable et dans la partie du code qui s'occupe de récupére les données de notre bases de données, dataSource sera instancié en tant qu'objet WorksheetDataTable :

'... Quelque part dans le code...
Dim dataSource As DataTable
Set dataSource = New WorksheetDataTable
ReadAndWriteData dataSource
    
'... Ailleurs dans le code...'
Dim dataSource As DataTable
Set dataSource = New MySQLDataTable
ReadAndWriteData dataSource

'... Quelque part dans le code...
Dim dataSource As DataTable
Set dataSource = New WorksheetDataTable
ReadAndWriteData dataSource
    
'... Ailleurs dans le code...'
Dim dataSource As DataTable
Set dataSource = New MySQLDataTable
ReadAndWriteData dataSource

Grâce à l'implémentation d'interface, nous avons séparé les responsabilités : la procédure ReadAndWriteData s'en contre fiche de savoir d'où viennent les données. Nous pouvons tout à fait rajouter une troisième classe FetchedAPIDataTable qui implémente l'interface DataTable, et le code de ReadAndWriteData reste le même !

L'héritage n'existe pas en VBA : est-ce un problème ?

La réponse est tout simplement non. Le seul mécanisme d'héritage du VBA est l'implémentation d'interface. Et c'est justement l'un des outils cruciaux dont on a besoin pour faire du code propre ! Oublions donc l'héritage entre classes et concentrons-nous sur la conception logicielle avec des interfaces.
